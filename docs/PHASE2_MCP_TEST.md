## Почему нужен GUI-шаг владельца
MCP Toolkit — внешняя обработка; её HTTP-сервер (порт 6003) стартует только когда обработка ОТКРЫТА в сессии 1С и нажата «Запустить сервер». Запустить это headless по SSH нельзя (толстому клиенту нужен рабочий стол). Поэтому — один аккуратный GUI-шаг владельца, дальше всё тестирует Claude по HTTP (curl с самой Винды, firewall не трогаем).

### 🔴 Ключевая находка (подтверждает ресёрч)
**Тумблеры инструментов в MCP Toolkit — косметика: они убирают инструмент из `tools/list` (discovery), но НЕ блокируют вызов.** Отключённый `execute_code` полноценно исполняет произвольный BSL-код при прямом `tools/call`. Доказано тремя разными телами (compile-ошибки на мой код + успешное `{"success":true,"data":706}`) и контролем (реальный отказ выглядит иначе).

**Следствие для безопасности (меняет план):** нельзя полагаться на тумблеры как на границу. Поскольку `execute_code` в серверном контексте без безопасного режима может вызвать `УстановитьПривилегированныйРежим(Истина)` и обойти любые роли (ресёрч), read-only гарантию строим ТОЛЬКО так:
1. **Форк обработки** — физически вырезать из `.epf` обработчики `execute_code` и прочих write-инструментов (нам для «второго мозга» нужен только `execute_query` + read-инструменты). GPL-3.0 позволяет. Это главный барьер.
2. **+ read-only пользователь `ai_reader`** — второй слой (на случай если что-то из read-инструментов окажется опаснее, чем кажется).
3. **+ безопасный режим** через профили безопасности кластера 1С — если оставляем клиент-серверный режим.
`execute_query` сам по себе безопасен (в языке запросов нет DML — подтверждено: читает, но конструкций записи не имеет).

### Статус сервера
На момент записи встроенный сервer ЗАПУЩЕН под сессией с полными правами (база без пользователей = админ), `execute_code` живой на `127.0.0.1:6003` (только localhost, под токеном, тестовая база с бэкапом). Разрушающих операций Claude не делал (только чтение). Перед перерывом — владельцу нажать «Остановить сервер» на форме.

## Стадия 2b — read-only ужесточение (ворота перед «боевым» использованием)
Стадия 2b пересмотрена после находки 2a: сначала **форк обработки** (вырезать execute_code и write-инструменты), потом read-only слой.
1. ✅ **Форк собран (2026-07-22)** — `windows/fork/` (патчер + BUILD.md). Заглушены обе функции исполнения кода (`ВыполнитьКод` сервер + `ВыполнитьКодНаКлиенте` клиент) → любой путь диспетчеризации execute_code возвращает «Неизвестный инструмент». Собрано headless конфигуратором (`/LoadExternalDataProcessorOrReportFromFiles`), выход `C:\1c\distr\MCP_Toolkit_FORK.epf` (1.83 MB, валидный контейнер 1С). Подтверждено на 3 уровнях (исходник/сборка/формат). Осталось: живой тест после переоткрытия форка владельцем (execute_code → отказ, execute_query → работает).
2. Создать пользователя `ai_reader` (Администрирование → Настройки пользователей и прав) с профилем «только просмотр/чтение», без администрирования и изменения.
3. Перезапустить обработку в сессии `ai_reader`; тесты §8: попытка записи через execute_query (языку нечем — подтвердить), запись любым способом → «Нарушение прав доступа», журнал регистрации без событий записи от ai_reader.
4. Escalation-тест: под ai_reader (если execute_code ещё доступен в каком-то виде) вызвать `УстановитьПривилегированныйРежим(Истина)` + попытку записи → должно блокироваться безопасным режимом/профилем.
