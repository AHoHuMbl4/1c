# План: «Второй мозг» компании на данных 1С:ERP

_Составлен 2026-07-22 по итогам веб-ресёрча (7 агентов: 4 ресёрчера + 3 адверсариальные проверки). Ключевые источники — в конце._

## 1. Цель и главный инвариант

Построить «второй мозг»: AI отвечает на произвольные вопросы по данным компании из 1С:ERP.

**Главное правило (инвариант проекта): 1С мы ТОЛЬКО читаем. Ни один запрос не должен ничего изменить в 1С.**

Из ресёрча следует принципиальный вывод: read-only гарантию нельзя строить на настройках MCP-инструмента (его защита — обходимый blacklist по ключевым словам, issue #14 у ROCTUP). Гарантия строится **на стороне самой 1С**, слоями, каждый из которых независим:

| Слой | Что даёт | Почему обязателен |
|---|---|---|
| 1. Выделенный пользователь 1С `ai_reader` с ролью «только чтение» (без прав Изменение/Добавление/Удаление/Проведение, **без права Администрирование**) | Платформа проверяет права всегда и на сервере; запись даст «Нарушение прав доступа» | Базовый барьер |
| 2. Только `execute_query` (язык запросов 1С) | Язык запросов **архитектурно не имеет DML** — операторов записи в нём нет (официальная позиция 1С); временные таблицы живут в tempdb/pg_temp и базу не трогают | Сужает поверхность до «нечем писать» |
| 3. Запрет выполнения произвольного кода (`execute_code` недоступен физически) | Любой серверный код ВНЕ безопасного режима может вызвать `УстановитьПривилегированныйРежим(Истина)` и обойти И роли, И RLS — **прав для этого не нужно** | Единственная реальная дыра read-only модели — закрываем её полностью |
| 4. «Защита от опасных действий» включена (платформа 8.3.9+) | Для внешних обработок, открытых интерактивно (Файл→Открыть), привилегированный режим не срабатывает | Стопор эскалации даже если код как-то выполнится |
| 5. Сетевой периметр: bind только на локальный/VPN-интерфейс, Bearer-токен, firewall | MCP Toolkit по умолчанию слушает 0.0.0.0 **без аутентификации** и без TLS | Никто чужой не дотянется до канала |
| 6. Бэкап перед каждым изменением на Винде (правило владельца) | `.dt`-выгрузка или копия файлов базы | Откат в любой момент |
| 7. Журналирование каждого запроса (на прокси + журнал регистрации 1С) | Всегда видно, кто что спрашивал | Аудит |

Что **не полагаться** (результаты адверсариальной проверки):
- Тумблеры отключения инструментов в MCP Toolkit документированы только как фильтр `tools/list` — call-time отказ не подтверждён. Прямой `tools/call` к «выключенному» инструменту нужно проверить на стенде; до проверки считаем тумблер мягким контролем.
- В режиме Python-прокси у ROCTUP отключения инструментов **нет вообще** и нет аутентификации эндпоинтов — прокси-режим в их виде не используем (либо только за нашим reverse proxy c auth).
- Blacklist ключевых слов (`DANGEROUS_KEYWORDS`) обходим принципиально (конкатенация строк, `Символ()`), фикс v1.8.0 не считаем полным.

## 2. Схема: предложение по изменению

Схема владельца: `текст → запрос 1С → получение → копирование в свою базу → анализ → ответ` — верная, но если ходить в 1С на каждый вопрос, мы (а) грузим рабочую ERP, (б) каждый раз рискуем каналом, (в) отвечаем медленно.

**Предложение — два контура, 1С трогаем минимально:**

```
                    ХОЛОДНЫЙ КОНТУР (ночью, по расписанию)
1С:ERP (Windows) ──[read-only канал]──► выгрузка порциями ──► «второй мозг» (LXC)
                                                                PostgreSQL+pgvector
                                                                + файлы Parquet/JSONL

                    ГОРЯЧИЙ КОНТУР (днём, на вопрос пользователя)
вопрос ──► LLM ──► SQL к СВОЕЙ базе (95% ответов, 1С не трогаем вообще)
              └──► если данных нет: точечный запрос 1С (язык запросов, read-only)
                   └──► результат тоже сохраняем к себе (база дорастает сама)
```

- Большинство ответов идёт из **своей** базы — быстро и с нулевым риском для 1С.
- В 1С ходим двумя способами: регулярная ночная инкрементальная выгрузка + редкие точечные ad-hoc запросы, когда нужного ещё нет. Каждый ad-hoc результат оседает у нас — база самообучается на реальных вопросах.
- Исторические снапшоты остаются у нас навсегда (1С хранит «сейчас», второй мозг — «как менялось») — это и есть основа для бизнес-решений.

## 3. Компоненты по машинам

**Windows `unde@10.8.0.58` (1С:ERP)** — минимум чужого кода:
- Пользователь 1С `ai_reader` (только чтение).
- Канал чтения (см. §4): MCP Toolkit (обработка в сессии `ai_reader`) и/или OData-публикация.
- Больше ничего. Все изменения — только после бэкапа.

**LXC `lxc-claude-1c` (Ubuntu, 192.168.56.42)** — всё остальное:
- PostgreSQL 16 + pgvector — основное хранилище (схемы: `raw` — jsonb-выгрузки как есть, `core` — нормализованные сущности, `search` — эмбеддинги).
- Каталог data lake: те же выгрузки в Parquet/JSONL (для SereneDB-эксперимента и переносимости).
- ETL-оркестратор (Python: расписание, пагинация, инкремент, upsert).
- MCP-прокси/валидатор запросов (наш, тонкий): журналирование, allowlist инструментов, rate limit.
- LLM-клиент (составление запросов, анализ) + позже бот-слой.

## 4. Канал чтения из 1С: что тестируем

**Кандидат A — 1C MCP Toolkit (ROCTUP), то что просил протестировать владелец.**
Плюсы: произвольные запросы на языке 1С (`execute_query`), метаданные (`get_metadata`), не меняет конфигурацию, не требует веб-сервера. Минусы: молодой проект, известные обходы blacklist, auth по умолчанию выключена, bind 0.0.0.0, TLS нет.
Жёсткая конфигурация для теста:
- версия ≥ 1.8.0; Bearer-токен обязателен; bind на VPN/локальный интерфейс + firewall Windows;
- сессия — строго под `ai_reader`; открытие интерактивно (Файл→Открыть) при включённой «защите от опасных действий»;
- тумблерами выключить: `execute_code`, `restart_1c_session`, `close_1c_session`, `get_screenshot`, `submit_for_deanonymization`; `ALLOW_DANGEROUS_WITH_APPROVAL=false`;
- **стендовая проверка обязательна**: прямой `tools/call execute_code` при выключенном тумблере должен получить отказ. Если не получает — форкаем toolkit (GPL-3.0, исходники открыты) и физически вырезаем опасные инструменты из сборки `.epf`. Это надёжнее любых тумблеров;
- их Python-прокси не используем.

**Кандидат B — стандартный OData (штатный механизм платформы).**
Плюсы: ничего стороннего в 1С вообще, права `ai_reader` полностью решают, состав публикации режется (`УстановитьСоставСтандартногоИнтерфейсаOData`), не-GET методы дополнительно режем на веб-сервере. Минусы: нужна публикация базы на веб-сервере (IIS) — админ-шаг на Винде; нет произвольных запросов языком 1С (только выборка сущностей с `$filter`); ~500–2000 строк/сек.
Роль: **основной канал холодного контура** (массовая ночная выгрузка), если публикация на IIS приемлема.
Бонус: существует готовый универсальный мост [OData→MCP (oisee/odata_mcp)](https://github.com/oisee/odata_mcp) — даёт MCP-интерфейс поверх штатного OData; если он зрел, получаем «MCP-удобство при OData-безопасности» вообще без стороннего кода в 1С. Проверить в фазе 2.

Рабочая гипотеза: **B для массовой выгрузки, A для произвольных ad-hoc запросов**. Если владелец не хочет IIS — всё делаем через A (execute_query умеет и массовую выборку, просто медленнее и через сторонний код).

## 5. Хранилище: SereneDB — да, но не сразу

Проверено ресёрчем + адверсариальной проверкой: SereneDB реальна (Берлин, open source Apache-2.0 с декабря 2025, PG-протокол, BM25 + векторный + гибридный поиск в одном SQL, zero-ETL чтение Parquet/JSON). Профиль — точно наш. **Но:** проекту публично ~8 месяцев, ядро — 3 инженера, прямо сейчас меняется storage-движок, ACID/транзакции не документированы, продакшн-референсов ноль.

Решение:
- **Основное хранилище — PostgreSQL + pgvector** (зрелость, ACID, JSONB; для объёмов 1С-выгрузок хватает с запасом).
- **SereneDB — экспериментальный слой** поверх того же data lake: она читает наши Parquet/JSONL напрямую (`read_parquet`), импорт не нужен, риска для основных данных ноль. Гоняем на ней гибридный поиск, сравниваем с pgvector. Пересмотр решения — когда их новый storage стабилизируется (~2027).

## 6. LLM и бот-слой

**DeepSeek (ключ получен, лежит в `credentials/deepseek.env`).**
⚠ Решение владельца: DeepSeek — облачный API, значит данные 1С (результаты запросов, документы) будут уходить на внешние серверы. Варианты:
1. **Компромисс (рекомендую на старт):** DeepSeek — только для *составления запросов* (видит имена метаданных: «Справочник.Номенклатура», реквизиты — но не данные) и для анализа *агрегатов* (суммы, счётчики). Сырые документы наружу не отдаём.
2. Полное доверие DeepSeek — всё в облако (просто, но данные компании наружу).
3. Полный self-host LLM на своём железе — максимальная приватность, нужен GPU.

**OpenClaw** (вопрос владельца) — open-source self-host ассистент-шлюз (ex-Clawdbot/Moltbot): каналы Telegram/WhatsApp, память, поддержка MCP-серверов, работает с OpenAI-совместимыми API (DeepSeek подходит). Как **бот-слой поверх** нашего MCP — хороший кандидат: получим «спросил в телеграме — получил ответ» почти бесплатно.
Но: (а) это ещё один автономный агент с доступом к данным ERP — ставить только на LXC, наружу не публиковать, инструменты ему давать только наши read-only; (б) в начале 2026 были громкие случаи открытых наружу инстансов — конфигурировать аккуратно. Решение: **в MVP не берём** (проверяем пайплайн простым клиентом — Claude Code / скрипт), подключаем в фазе 5, когда канал и база устоятся.

## 7. Фазы работ

**Фаза 0 — доступы и инвентаризация** (ничего не меняем):
SSH до LXC и Винды; версия платформы 1С (нужна 8.2.13+/8.3.25 для toolkit), вариант базы (файловая/клиент-серверная — влияет на профили безопасности), СУБД, есть ли IIS, объёмы данных; механика бэкапа на Винде (куда, сколько места). → заполняем `techContext.md`.
✔ Чек-поинт: обе машины доступны, факты записаны.

**Фаза 1 — фундамент на LXC:**
PostgreSQL+pgvector, схемы `raw`/`core`/`search`, каталог data lake, Python-окружение, наш тонкий прокси-журналировщик.
✔ Чек-поинт: локальный smoke — вставка/выборка/векторный поиск.

**Фаза 2 — read-only канал в 1С** (первые изменения на Винде, каждое — после бэкапа):
бэкап базы → пользователь `ai_reader` → MCP Toolkit по жёсткой конфигурации §4 → **тесты безопасности** (см. §8) → при желании владельца параллельно OData.
✔ Чек-поинт: `execute_query` работает под `ai_reader`, ВСЕ тесты безопасности зелёные.

**Фаза 3 — MVP пайплайна (горячий контур):**
кэш метаданных ERP у нас → текст → LLM составляет запрос 1С → валидация (только выборка, лимиты) → выполнение → сохранение в `raw` → анализ → ответ. Канонический тестовый вопрос типа «продажи за прошлую неделю по топ-10 номенклатуры».
✔ Чек-поинт: 5 разных вопросов отвечены end-to-end, каждый запрос в журнале, в 1С — ноль изменений (проверка по журналу регистрации).

**Фаза 4 — холодный контур:**
ночная инкрементальная выгрузка приоритетных разделов (документы — по дате с перезаписью «хвоста» N дней, справочники — полной ночной перезаливкой; `ВерсияДанных` как курсор не используем — ненадёжна); нормализация в `core`, эмбеддинги в `search`.
✔ Чек-поинт: утром данные за вчера у нас; повторный прогон идемпотентен.

**Фаза 5 — надстройки:**
бот-слой (OpenClaw) на LXC; SereneDB-эксперимент над data lake; расширение покрытия разделов ERP.

## 8. Тесты безопасности (ворота фазы 2 → 3)

Перед первым «боевым» вопросом — все обязаны быть зелёными:
1. Запись через `execute_query` — невозможна синтаксически (языку нечем писать) — подтверждаем попыткой.
2. Прямой `tools/call execute_code` при выключенном тумблере → отказ. Иначе — форк и вырезание из сборки.
3. Код с `УстановитьПривилегированныйРежим(Истина)` (если execute_code вообще достижим) → заблокирован «защитой от опасных действий».
4. Попытка записи объекта от `ai_reader` любым способом → «Нарушение прав доступа».
5. Запрос без Bearer-токена → 401; запрос с чужого интерфейса → соединение не устанавливается.
6. OData (если включаем): POST/PATCH/DELETE → отказ (права + веб-сервер).
7. После всей серии — журнал регистрации 1С: ни одного события записи от `ai_reader`.
8. Нагрузочный: тяжёлый запрос в рабочее время не блокирует пользователей (версионное чтение), но правило по умолчанию — тяжёлое только ночью.
