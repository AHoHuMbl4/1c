# SereneDB — движок отчётов/аналитики над 1С (решение владельца 2026-07-23)

> Владелец знает фаундеров лично, доверяет команде → строим витрину/аналитику на **SereneDB**.
> Установка/раскатка — `ubuntu/serenedb/README.md`. Бот-слой — `docs/OPENCLAW_BOT.md`.

## Что это и зачем
SereneDB (Apache-2.0 single-node, Berlin 2025) — одна БД, объединяющая то, что мы иначе
стыковали бы руками: **полнотекст + вектор (HNSW/BM25) + колоночный OLAP** (Velox; под капотом
DuckDB) с real-time обновлениями, всё через **PostgreSQL-протокол**. Проверено локально: коннект
`psql`, представляется как `PostgreSQL 18.3 (SereneDB 26.07.3)`, агрегации GROUP BY считают верно.

Закрывает наши три задачи разом:
1. **Витрина-аналитика** (колоночный OLAP) → скорость на объёме, точные суммы/обороты по ВСЕМ строкам.
2. **Семантический резолвер** (vector+fuzzy) → 2-этап «найти нужную сущность по смыслу → точно
   посчитать», можно одним гибридным запросом (см. `OPENCLAW_BOT.md` про NL→запрос).
3. **Свежесть** — real-time ingestion.
Postgres-фронт → низкий порог входа (у нас уже Postgres/pgvector; переезд почти drop-in).

## Почему риск молодой БД у нас ОГРАНИЧЕН
SereneDB держит **производную витрину, пересобираемую из 1С в любой момент**. Источник истины —
1С (read-only, не трогаем). Даже полная потеря витрины = пересинхрон из 1С, данные не теряются.
Поэтому ставить на неё аналитику/поиск — низкий риск (это НЕ система-оф-рекорд).

## Как развёрнуто (.42)
Сервис `serenedb` (systemd, **enabled = reboot-safe**), бинарь `/usr/local/bin/serened`, конфиг
`/etc/serenedb/serened.conf`, данные `/var/lib/serenedb`, под юзером `serened` (не root).
**Только loopback `127.0.0.1:7890`** — наружу не выставлено. Версия 26.07.3.

## Безопасность
- Слушает только loopback; доступ локальный (как Postgres:5432 на этом же боксе).
- Есть `--hba_config` (pg_hba-стиль), `--auth_api_key/bearer`, TLS — включим при выносе за хост.
- Инвариант 1С read-only не затронут: SereneDB читает выгрузку, к 1С не ходит.

## Реальность данных 1С-базы (замер OData 2026-07-23) 🔴 ВАЖНО
Проба служебного документа OData вскрыла главное про источник:
- Подключённая база — **полноценная конфигурация** (ERP-класса): **2968 сущностей** (1419 справочников,
  1549 документов). Но **бизнес-данных в ней почти нет**: `Catalog_Номенклатура`=0, `Контрагенты`=2,
  `Организации`=1; **все документы = 0** (проверено 60 верхнеуровневых: продажи/поступления/платежи/касса
  — пусто). Единственная населённая таблица — встроенный `Catalog_КлассификаторБанков` (2779).
  → Поэтому «банки в Казани» работают, а «продажи по месяцам» пока невозможны: **данных нет**, не код виноват.
- **Регистры (Accumulation/Accounting/Information/Calculation) в OData НЕ опубликованы — 0 по префиксу.**
  А обороты/остатки/суммы в 1С живут именно в регистрах → **готовых агрегатов через OData нет**.
- Список синка `1c-etl-selected.txt` содержит имена не из этой конфигурации (отсюда 404 в синке):
  `Catalog_Договоры` → реально `Catalog_ДоговорыКонтрагентов`; плоских `ЕдиницыИзмерения`/`Кассы` нет
  (есть `Catalog_КлассификаторЕдиницИзмерения`; каталога касс нет). Файл, возможно, общий с braine ETL —
  **не трогал**; чинить имена у источника выбора.

**Вывод:** аналитический стек (SereneDB + NL→SQL + резолвер + гейт + графики) готов и корректен, но
упирается в ДАННЫЕ, а не в код. Для реальной аналитики нужны две вещи на стороне 1С (решение владельца):
(1) база с реальными данными (эта — пустая); (2) публикация нужных `AccumulationRegister` (продажи,
выручка, ДС…) в стандартный интерфейс OData 1С — тогда SereneDB считает обороты по периодам. Документы
даже заполненные дают строки, а не готовые проводки/обороты.

## Защита от рассинхрона выбора сущностей с OData (без хардкода)
**Урок:** в списке синка `1c-etl-selected.txt` 3 из 23 имён были написаны «как обычно в 1С» и не
сверены с этой конфигурацией → молчаливый 404, висели незамеченными. Наш косяк — не сами имена, а то,
что пайплайн их **тихо глотал**. Принцип-лекарство: **правда об именах берётся из живого OData, а не из
предположений; ручной выбор валидируется против источника и подсказывается семантикой.** Никаких
имён-констант.
- [x] **Слой 1 — преполёт-сверка (`serene_sync._preflight`).** Перед загрузкой тянем служебный документ
      OData (`poc_load_entity.published_entity_sets`) и сверяем весь выбор. Несуществующее имя →
      **громко** + прогон помечается failed (`sys.exit(3)` → systemd покажет unit failed, не упустить).
      Валидные сущности при этом грузятся; невалидные пропускаются (не долбим заведомый 404).
- [x] **Слой 2 — семантическая подсказка.** Для мёртвого имени показываем ближайшее реальное
      (лексический fuzzy, тот же принцип, что в резолвере): `Договоры → ДоговорыКонтрагентов`. **Не
      подменяем молча** — только показываем (иначе новый «понял не так»).
- [ ] **Слой 3 — выбор ИЗ живого списка (структурный, ещё не сделан — НЕ УПУСТИТЬ).** Сам выбор
      сущностей строить не рукописно, а **из реального служебного документа OData** (галочки по
      фактически опубликованным именам). Тогда несуществующее имя **нельзя выбрать в принципе** — ошибка
      исчезает по построению, а не ловится постфактум. Это убирает и класс «config обновился, сущность
      переименовалась». Затрагивает UX выбора (возможно, общий с braine ETL — согласовать источник).
      Тот же принцип шире списка: пайплайн выводит структуру из источника (интроспекция), а не из
      предположений; где нужен ручной ввод — валидировать против источника.

## Backup/restore (проверено 2026-07-23)
- **Стратегия:** первичное восстановление витрины — **ре-синк из 1С** (`serene_sync`), т.к. витрина
  ПРОИЗВОДНАЯ (не система-оф-рекорд). Быстрый restore без пула OData/ре-эмбеддинга — **оффлайн-снапшот**
  `/var/lib/serenedb` (`ubuntu/serenedb/serenedb_backup.sh`, ротация KEEP).
- **Проверено round-trip:** снапшот → дроп таблицы → restore из снапшота → данные вернулись (маркер
  777), витрина+резолвер целы. Онлайн-пути: `EXPORT DATABASE` только в writable-подпапку (sandbox
  ProtectSystem); `pg_dump` не годится (version mismatch — SereneDB не полноценный PG).

## Открытые вопросы фаундерам (контакт владельца)
- Штатный online-backup/PITR single-node (кроме снапшота/EXPORT) — есть ли?

## Инвариант
Отчёт/график достоверен **по построению**: числа считает наш SQL из реальных данных, не LLM.
LLM только (а) понимает вопрос и (б) оживляет текст — а его гейт (`verify-plugin`) сверяет.
Новый класс риска здесь — не «выдумал число», а «понял вопрос не так» → страховка = прозрачность
трактовки (метрика/разрез/период/фильтры рядом с ответом) + сверка итогов с родным отчётом 1С.
