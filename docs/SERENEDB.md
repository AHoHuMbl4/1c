# SereneDB — движок аналитики/отчётов над 1С

Аналитический слой «второго мозга»: превращает вопрос на естественном языке в точный отчёт/график по
данным 1С. **Общий, без хардкода — переносится на любую 1С-базу и локаль.** 1С только читаем (через OData).

- Установка движка SereneDB: `ubuntu/serenedb/README.md`.
- **Развёртывание на новом сервере / подключение новой 1С-базы: `docs/RUNBOOK_DEPLOY.md` §10.**
- Бот-слой (Telegram, гейт анти-галлюцинаций): `docs/OPENCLAW_BOT.md`.
- План доведения до продакшена и что осталось: `docs/PRODUCTION_PLAN.md`.

## Что это
SereneDB (Apache-2.0, single-node, **PostgreSQL-протокол**, под капотом форк DuckDB) — БД, объединяющая
полнотекст + вектор + колоночный OLAP. У нас держит **производную витрину**, пересобираемую из 1С в любой
момент. Источник истины — 1С; даже полная потеря витрины = ре-синк, данные не теряются. Поэтому строить на
ней аналитику — низкий риск (это НЕ система-оф-рекорд).

## Пайплайн (сквозной, общий)
```
1С ─OData(GET,read-only)─► ЗАГРУЗЧИК ─► витрина SereneDB ─► NL→SQL ─► ВАЛИДАТОР ─► выполнение(serene_ro) ─► отчёт+график ─► бот
     poc_load_entity.py      catalog_*/document_*   serene_report.py   allow-list(AST)      + caveat подмены      report_1c(MCP)
```
1. **Загрузчик** (`poc_load_entity.py`) — ЛЮБАЯ сущность 1С через OData → таблица `catalog_*`/`document_*`
   с сырыми колонками (union полей, авто-типы). Инварианты, общие для любой базы:
   стабильная пагинация `$orderby=Ref_Key` (иначе страницы OData перекрываются → дубли/потери); дедуп
   «одна строка на объект 1С» (`QUALIFY row_number() PARTITION BY ref_key`); исключение строк-папок
   (`IsFolder` — узлы иерархии справочника, не бизнес-строки).
2. **Схема для LLM** (`get_schema`) — интроспекция `duckdb_columns()` + **размеры таблиц** (`[строк: N]`,
   чтобы LLM отличал населённую таблицу от почти пустой при похожих именах) + **примеры значений**.
3. **Резолвер** (`resolve_hints`) — фаззи-термин вопроса → ТОЧНОЕ значение колонки-измерения. Два
   **язык-нейтральных** слоя (без стоп-слов/стемминга под язык): лексический (подстрока+`jaro_winkler`) и
   семантический (эмбеддинги Qwen `text-embedding-v4`@1536 → `resolver_index`, косинус, порог 0.70).
4. **gen_sql** (DeepSeek, `temperature=0`, промт язык-нейтральный) — вопрос+схема+подсказки → один SELECT.
5. **Валидатор** (`validate`) — см. «Безопасность».
6. **Выполнение** под ролью `serene_ro` → результат + `sql` (прозрачность трактовки) + `caveat` при
   подмене метрики. График — `render_chart` (PNG).
7. **Бот** — инструмент `report_1c` (MCP `1c-mcp-reports`, :6015); гейт `verify-plugin` сверяет числа.

## Выбор сущностей (имена — из реальности, не из головы)
- `serene-entities.txt` — активный список; **собственный у serene** (версионируется в git, деплоится с
  кодом). НЕ `/etc/1c-etl-selected.txt` (тот — у braine/ETL; разные системы, совпадать не обязаны).
- `serene_select.py` тянет служебный документ OData и пишет список из РЕАЛЬНЫХ имён; `--review` — picker:
  кандидаты закомментированы по убыванию строк, владелец отбирает бизнес-сущности (авто-дискавери на 1С
  неизбежно включает и платформенные справочники — метаданные/классификаторы; отделить их алгоритмически
  без хардкода нельзя, поэтому финальный отбор — за человеком).
- `serene_sync._preflight` каждый прогон сверяет весь список с живым OData: несуществующее имя → **громко**
  (прогон помечается failed) + подсказка ближайшего реального, а не молчаливый 404.

## Безопасность (защита в глубину)
- **Валидатор = ALLOW-LIST через AST** (`json_serialize_sql`): пропускается ТОЛЬКО одиночный SELECT/WITH,
  читающий ИСКЛЮЧИТЕЛЬНО таблицы витрины (+ свои CTE). DML/DDL/несколько-операторов → `error` парсера →
  reject. Табличные функции (`read_csv`/`glob`/…) и объекты вне схемы (`resolver_index`, `pg_settings`,
  системные каталоги) → reject *by construction*. Плюс лимит длины SQL (анти-DoS) и тонкий денайлист
  скалярных служебных функций (`current_setting`). Общий контроль **без списков имён** — на любую базу.
  (Денайлисты `FORBIDDEN`/`FS_ACCESS`/`INTERNAL_OBJECTS` остаются как fallback, если AST недоступен.)
- **Роли БД** (защита в глубину под валидатором):
  - `serene_ro` — read-only: SELECT работает, любая запись → `permission denied` (проверено `test_ro_role`).
  - `serene_resolver` — читает служебный `resolver_index` (positive control); у `serene_ro` доступ к нему
    ОТОЗВАН, поэтому LLM-SELECT не дотянется к внутреннему индексу даже в обход валидатора.
- **1С read-only** — только OData GET (гарантия на стороне 1С/шлюза, см. RUNBOOK §6).
- ⚠ **Движковый лимит доступа к ФС недостижим** на текущей SereneDB (один общий DuckDB-инстанс, FS-настройки
  `enable_external_access`/`allowed_directories` — GLOBAL-only, ломают загрузчик; подтверждено исходником
  SereneDB). Поэтому доступ к файлам держит ТОЛЬКО валидатор (allow-list). Это пробел модели SereneDB →
  вопрос фаундерам (ниже).

## Достоверность (инвариант)
Отчёт/график достоверен **по построению**: числа считает наш SQL из реальных данных, не LLM. LLM только
(а) понимает вопрос, (б) оживляет текст. Класс риска аналитики — не «выдумал число», а **«понял вопрос не
так»** и **«подмена метрики»**; страховка:
- **Прозрачность трактовки** — рядом с ответом виден `sql` (владельцу в логе), метрика/разрез/фильтры.
- **Подмена метрики** закрыта общим кодом: `measure_caveat` (типовой — числовая свёртка ТЕКСТ-реквизита,
  напр. `SUM(CAST(номер_счёта AS ...))`, по типам из интроспекции) + `metric_critic` (LLM-проход: «есть ли
  реальная колонка-величина под запрос?»). Не блок, а **предупреждение клиенту**.
- **Гейт бота** (`verify-plugin`) сверяет числовые токены исходящего с эталоном инструмента.

## Реальность данных / что нужно для реальной аналитики
Аналитический стек готов и общий, но упирается в ДАННЫЕ, а не код. Для реальной аналитики (обороты/суммы/
периоды) нужны две вещи **на стороне 1С** (решение владельца):
1. База с реальными транзакциями (документы/движения).
2. **Публикация нужных `AccumulationRegister`** (продажи, выручка, ДС) в стандартный интерфейс OData 1С —
   иначе готовых оборотов/остатков через OData нет (документы дают строки, а не проводки/обороты).
Пока таких данных нет, агрегации по периодам/суммам физически не на чем проверить — это гейт продакшена
(`PRODUCTION_PLAN.md` §7). Агент-харнесс готов прогнать реальные обороты, как появятся данные.

## Backup/restore
- Первичное восстановление — **ре-синк из 1С** (`serene_sync`), т.к. витрина производная.
- Быстрый restore без ре-пула/ре-эмбеддинга — **оффлайн-снапшот** `/var/lib/serenedb`
  (`serenedb_backup.sh`, ротация). Проверено round-trip. `pg_dump` не годится (SereneDB не полноценный PG).

## Тесты (гейты; гоняются перед деплоем/после синка)
`test_validate` (allow-list), `test_integrity` (grain / витрина⊆OData / resolver / нет фантомных строк),
`test_caveat` (подмена метрики), `test_ro_role` (read-only + positive control), `test_e2e` (реальный
`run_report`: ground-truth **из БД** + danger-wiring). Все — схемо-агностичны (таблицы/числа из данных, не
вшиты). Запуск — в окружении reports (env как у `probe.sh`). Плюс агент-харнесс (5 классов провокаций).

## Открытые вопросы фаундерам SereneDB (контакт владельца)
- Ограничение доступа к ФС **per-role/per-connection** (сейчас `enable_external_access`/`allowed_directories`
  — GLOBAL-only, ломают загрузчик; как запретить read_csv/glob только отчётной роли?).
- Штатный online-backup/PITR single-node (кроме снапшота/`EXPORT DATABASE`).
- Граница open-source/commercial; HNSW-индекс (в этой версии `CREATE INDEX USING HNSW` нет — сейчас перебор
  косинуса, для больших справочников нужен индекс/новая версия).
- Прод-готовность single-node, durability под нагрузкой; реальная скорость на объёме (замер, не маркетинг).
