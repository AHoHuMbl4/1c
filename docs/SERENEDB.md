# SereneDB — движок отчётов/аналитики над 1С (решение владельца 2026-07-23)

> Владелец знает фаундеров лично, доверяет команде → строим витрину/аналитику на **SereneDB**.
> Установка/раскатка — `ubuntu/serenedb/README.md`. Бот-слой — `docs/OPENCLAW_BOT.md`.

## Что это и зачем
SereneDB (Apache-2.0 single-node, Berlin 2025) — одна БД, объединяющая то, что мы иначе
стыковали бы руками: **полнотекст + вектор (HNSW/BM25) + колоночный OLAP** (Velox; под капотом
DuckDB) с real-time обновлениями, всё через **PostgreSQL-протокол**. Проверено локально: коннект
`psql`, представляется как `PostgreSQL 18.3 (SereneDB 26.07.3)`, агрегации GROUP BY считают верно.

Закрывает наши три задачи разом:
1. **Витрина-аналитика** (колоночный OLAP) → скорость на объёме, точные суммы/обороты по ВСЕМ строкам.
2. **Семантический резолвер** (vector+fuzzy) → 2-этап «найти нужную сущность по смыслу → точно
   посчитать», можно одним гибридным запросом (см. `OPENCLAW_BOT.md` про NL→запрос).
3. **Свежесть** — real-time ingestion.
Postgres-фронт → низкий порог входа (у нас уже Postgres/pgvector; переезд почти drop-in).

## Почему риск молодой БД у нас ОГРАНИЧЕН
SereneDB держит **производную витрину, пересобираемую из 1С в любой момент**. Источник истины —
1С (read-only, не трогаем). Даже полная потеря витрины = пересинхрон из 1С, данные не теряются.
Поэтому ставить на неё аналитику/поиск — низкий риск (это НЕ система-оф-рекорд).

## Как развёрнуто (.42)
Сервис `serenedb` (systemd, **enabled = reboot-safe**), бинарь `/usr/local/bin/serened`, конфиг
`/etc/serenedb/serened.conf`, данные `/var/lib/serenedb`, под юзером `serened` (не root).
**Только loopback `127.0.0.1:7890`** — наружу не выставлено. Версия 26.07.3.

## Безопасность
- Слушает только loopback; доступ локальный (как Postgres:5432 на этом же боксе).
- Есть `--hba_config` (pg_hba-стиль), `--auth_api_key/bearer`, TLS — включим при выносе за хост.
- Инвариант 1С read-only не затронут: SereneDB читает выгрузку, к 1С не ходит.

## Открытые вопросы (в т.ч. фаундерам — контакт владельца)
- **Backup/restore** штатно в доке не описан → протестировать (снапшот `/var/lib/serenedb`?) и
  спросить фаундеров. Смягчено тем, что витрина — производная.
- Граница open-source / commercial (HA, scale-out) — уточнить.
- Прод-готовность single-node, durability под нагрузкой — замер + честный ответ команды.
- Реальная скорость на нашем объёме — только замер, не маркетинг (трактовку «trust but verify»
  применяем как к любым вендор-цифрам).

## Инвариант
Отчёт/график достоверен **по построению**: числа считает наш SQL из реальных данных, не LLM.
LLM только (а) понимает вопрос и (б) оживляет текст — а его гейт (`verify-plugin`) сверяет.
Новый класс риска здесь — не «выдумал число», а «понял вопрос не так» → страховка = прозрачность
трактовки (метрика/разрез/период/фильтры рядом с ответом) + сверка итогов с родным отчётом 1С.
