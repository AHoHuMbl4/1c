# Почему встроенный сервер MCP Toolkit зависает — точная причина (из кода)

_2026-07-22. Разобрано по исходникам тулкита (ROCTUP/1c-mcp-toolkit): нативная компонента `native_components/MCPHttpTransport/src/http_transport.cpp` + модуль формы `Forms/Форма/.../Module.bsl`. Не гипотеза — механизм из кода._

## Как устроено обслуживание запросов

```
HTTP-клиент → [нативка cpp-httplib, отдельный поток] → ВнешнееСобытие "MCP_POST" в 1С
                     │ поток БЛОКируется (WaitForResponse), ждёт 1С
                     ▼
        [1С idle-обработчик «ЦиклОпроса», ~1 сек] → достаёт ОДНУ команду за цикл,
                     выполняет (execute_query…), отдаёт ответ обратно в нативку
                     ▼
        нативка будит заблокированный поток → ответ уходит клиенту
```

Факты из кода:
- **Каждый** HTTP-эндпоинт (`/mcp`, `/api/*` и даже **`/health`**) обработчик регистрирует как `HandleRequest(...)` → шлёт событие в 1С и `WaitForResponse` **блокирует рабочий поток httplib** до ответа 1С (`http_transport.cpp:47-77, 230`). У нативки **нет ни одного самостоятельного ответа** — всё ждёт 1С.
- Ответы формирует **клиентский обработчик ожидания** `ПодключитьОбработчикОжидания("ЦиклОпроса", 1, Ложь)` (`Module.bsl:736`). Тело `ЦиклОпроса` (`Module.bsl:824`) за один вызов берёт **ровно ОДНУ** команду: `Если РезультатЗапроса.ЕстьКоманда Тогда ОбработатьКоманду(...)`.
- Пул рабочих потоков cpp-httplib маленький (дефолт ≈ ядра−1), явного увеличения в коде нет.

## Два режима отказа (оба воспроизведены)

1. **Пачка/конкурентность.** N одновременных запросов блокируют N потоков httplib; 1С разбирает их по одному за ~1 сек. Лишние соединения встают в очередь, клиенты не дожидаются и рвут связь → на тулките сокеты копятся в **CLOSE_WAIT**. Когда блокированы все потоки пула — сервер перестаёт принимать вообще что-либо, включая `/health`.
2. **🔴 Модальный диалог (killer для прода).** Idle-обработчики 1С **не срабатывают, пока в сессии открыт модальный диалог** (предупреждение, «защита от опасных действий», ошибка, любое окно) или сессия занята. Тогда `ЦиклОпроса` не крутится → все входящие запросы висят вечно → пул потоков исчерпан → сервер полностью «мёртв» до закрытия диалога. Это объясняет, почему даже одиночный последовательный запрос повисает: в сессии открыт диалог.

Пропускная способность даже в идеале — **~1 запрос/сек** (гранулярность idle-обработчика, одна команда за цикл).

### Подтверждено вживую (2026-07-22)
На стенде тулкит завис именно из-за модального окна Бухгалтерии **«Проверка правомерности использования конфигурации»** (просит подключить Интернет-поддержку ИТС; кнопки «Ввести сейчас/позже»). Пока оно открыто — `ЦиклОпроса` стоит, тулкит не отвечает даже на `/health`. Диалог **периодический** (проверка легальности конфигурации БСП) → будет вешать встроенный сервер регулярно. Лечение на стенде — «Ввести позже» при появлении; на прод это не переносится (ещё один довод за OData, который клиентские диалоги не трогают).

## Вывод для продакшна

**Встроенный сервер тулкита НЕ годится для надёжного прода.** Он завязан на однопоточный клиентский idle-обработчик 1С, который: (а) отдаёт ~1 запрос/сек, (б) полностью встаёт при любом модальном окне в сессии. Для интерактивной отладки/одиночного юзера — ок; для сервиса «второго мозга» — нет.

### Что делаем
- **Прод-канал чтения — серверный, без idle-обработчика:** штатный **OData** 1С, опубликованный на веб-сервере (IIS/Apache) — он многопоточный, обслуживается сервером/rphost, не зависит от модальных окон сессии; read-only обеспечивается правами пользователя. Наш whitelist-шлюз ставится перед ним так же. (Это Кандидат B из `docs/PLAN.md`.) Альтернатива — опубликованный HTTP-сервис (нужно расширение конфигурации).
- **Для текущего MVP/теста** оставляем встроенный тулкит, но:
  - наш `gateway.py` **сериализует** обращения (1 конкурентный) + `Connection: close` → снимает режим отказа №1 (исчерпание пула от конкурентности);
  - режим отказа №2 (модальный диалог) кодом шлюза НЕ лечится — **сессия 1С с тулкитом должна быть строго без открытых диалогов** (выделенная сессия, ничего модального сверху). Даже так — потолок ~1 запрос/сек.
- Итог: тестировать пайплайн на встроенном тулките можно (сессия чистая, запросы последовательные), но переносить на прод — только на OData/HTTP-сервис.

## Как разбудить зависший тулкит сейчас
На Винде в сессии 1С: закрыть ЛЮБОЙ открытый диалог/окно поверх формы тулкита; если не помогло — на форме «Остановить сервер» → «Запустить сервер». После рестарта не слать пачками — только через наш сериализующий шлюз.
