# Архитектура «второго мозга» на данных 1С

Полное устройство системы: компоненты, потоки данных, безопасность, эксплуатация.
Как раскатать с нуля — [`RUNBOOK_DEPLOY.md`](RUNBOOK_DEPLOY.md).

Проверено end-to-end 2026-07-22..23 на стенде: Windows 11 + платформа 1С 8.3.27.1786 (x86) +
Бухгалтерия 3.0 → OData/IIS → Ubuntu LXC → бот в Telegram отвечает по данным 1С: **факты** (braine
RAG, с цитатами) и **аналитику/отчёты** (SereneDB, таблица + график), живым тоном (OpenClaw).

Слои документированы отдельно: бот-слой — [`OPENCLAW_BOT.md`](OPENCLAW_BOT.md); аналитика/витрина —
[`SERENEDB.md`](SERENEDB.md). Этот файл — вся картина и то, как слои связаны.

---

## 1. Цель и инвариант

Дать AI-боту доступ к данным компании из 1С, чтобы он отвечал на вопросы бизнеса.

🔴 **Инвариант: 1С только читаем.** Ни один запрос не должен ничего изменить в 1С. Гарантия — слоями (§4), не на честном слове.

Замысел: данные 1С регулярно выгружаются во внешние хранилища; бот отвечает по ним, а не дёргает
1С на каждый вопрос (и не под риском записи). Два «мозга» по типу вопроса:
- **braine (RAG)** — факты и «найди по смыслу» (текст, договоры, реквизиты), ответ с цитатами.
- **SereneDB (витрина)** — точные агрегации/отчёты/графики по ВСЕМ строкам (NL→SQL), под read-only ролью.

Сверху — **OpenClaw** (диалоговая оболочка тона, Telegram-бот): переформулирует факты мозгов живо,
сам данные не трогает. 🔴 **2-й инвариант (правило владельца): правила/ограничения — КОДОМ, не
промтом.** Анти-галлюцинации (сверка чисел), анти-слив внутреннего, read-only — детерминированным
кодом/гейтом/ролью; промт (персона) держит только тон. Эмбеддинги — только Qwen `text-embedding-v4`.

## 3. Компоненты и потоки данных

### Windows (штатная 1С, ничего кастомного)
- **1С:база** — файловая (стенд) или клиент-серверная (прод).
- **IIS** (служба Windows) публикует базу; включён штатный **OData** (`/1c/odata/standard.odata/`). Состав OData (какие объекты отдавать) задаётся один раз при публикации перебором всех справочников/документов.
- **Пользователь `ai_reader`** — роль «Только просмотр» (read-only). OData-запросы идут под ним (Basic auth).

### Ubuntu LXC (наш контроль)
**Наполнение из 1С:**
| Сервис | Порт | Роль |
|---|---|---|
| `1c-odata-gateway` | 6011 | Read-only шлюз: только GET к OData; POST/PATCH/PUT/DELETE → 405. Хранит креды 1С. |
| `1c-config-ui` | 6012 | Веб-галочки «что тянуть» → `/etc/1c-etl-selected.txt`. |
| `1c-etl.timer/.service` | — | Ночью (03:00) выгружает выбранные сущности через шлюз → md-таблицы → KB-репо (GitLab). |

**braine (RAG-мозг — факты/текст):**
| Сервис | Порт | Роль |
|---|---|---|
| `open-webui` | 3000 | Индекс/retrieval. Эмбеддинги **Qwen `text-embedding-v4` @ 1536** (DashScope), pgvector 1536. |
| `oikb` | 8081 | Синк KB-репо (GitLab) → Open WebUI. |
| `kb-poll` | — | Новый коммит в KB-репо → триггер синка oikb. |
| `rerank-shim` | 8082 | Реранкер qwen3-rerank (DashScope). |
| `api` | 8090 | HTTP API `POST /ask` (retrieval → гейты точности → DeepSeek → ответ с цитатами). |
| `postgresql` | 5432 | pgvector (вектора OWUI) + служебные схемы braine. |
| ~~`tg-bridge`~~ | — | **Выведен из строя (disabled):** Telegram забрал OpenClaw. braine теперь фронт не держит, только `/ask`. |

**SereneDB (аналитика/отчёты):**
| Сервис | Порт | Роль |
|---|---|---|
| `serenedb` | 7890 | Витрина: колоночный OLAP + вектор, Postgres-протокол. Loopback, юзер `serened`. Бот — под read-only ролью `serene_ro`. |

**Бот-слой (OpenClaw, диалог):**
| Сервис | Порт | Роль |
|---|---|---|
| `1c-mcp-braine` | 6014 | MCP-инструмент `ask_1c` над braine `/ask` (факты/текст). |
| `1c-mcp-reports` | 6015 | MCP-инструмент `report_1c`: NL→SQL по SereneDB (под ro-ролью) + резолвер Qwen + график PNG. |
| `openclaw-gateway` (user) | 18800 | Движок OpenClaw (юзер `undebot`): Telegram `@test1c_mcp_bot`, DeepSeek-тон, verify-плагин (гейт кодом). |
| `1c-bot-monitor.timer` | — | Каждые 3 мин: жив ли gateway/serenedb/mcp/api → прямой Telegram-алерт владельцу при падении. |

### Холодный контур (наполнение, ночью)
```
1С OData ──GET── шлюз :6011 ── ETL: выгрузка выбранных сущностей → md-таблицы
   (резолв ссылок guid→наименование, чистка полей) →
   ├─► git push в KB-репо (GitLab) → kb-poll → oikb → OpenWebUI индексирует (Qwen-эмбеддинги) → pgvector   [braine RAG]
   └─► SereneDB-витрина (сейчас POC-загрузчик; штатный инкремент-синк — в очереди)                          [аналитика]
```

### Горячий контур (диалог)
```
человек → Telegram @test1c_mcp_bot → OpenClaw (тон, DeepSeek) — сам выбирает инструмент:
   ├ report_1c → SereneDB: NL→запрос (LLM понимает намерение) → read-only SELECT (валидатор+ro-роль)
   │             + резолвер (термин→точное значение, Qwen-вектор) → таблица + график PNG
   └ ask_1c    → braine: retrieval (BM25+вектор, реранк) → гейты точности → ответ с цитатами
   → verify-плагин (ГЕЙТ, КОДОМ): числа ответа сверяет с данными инструмента (несверенное режет/
       заменяет); режет внутреннее (SQL/пути) на исходящем и в подписи к фото
   → живой ответ клиенту (таблица/график); «нет данных» — честно, не выдумывает
```
Точность и безопасность здесь дают **код**, не промт: braine-гейты + `verify-плагин` +
read-only роль SereneDB. Персона (тон, маршрутизация инструментов) — промт, но НЕ слой гарантий.
Детали бот-слоя — [`OPENCLAW_BOT.md`](OPENCLAW_BOT.md); аналитики — [`SERENEDB.md`](SERENEDB.md);
стек braine — [`UBUNTU_SETUP.md`](UBUNTU_SETUP.md).

## 4. Безопасность (read-only + анти-галлюцинации кодом)

**1С — только чтение, два независимых слоя** (для OData):
1. **Пользователь `ai_reader`** — прав записи нет: любая попытка записи через OData → отказ (проверено: чтение 200, запись 500).
2. **OData-шлюз** — deny-by-default по методу: только GET проходит; POST/PATCH/PUT/DELETE → 405, до 1С не доходят (проверено с LXC).

Плюс: **язык OData на чтение** (GET) не меняет данные по построению; состав OData ограничен опубликованными объектами; креды 1С — только в шлюзе (`/etc/1c-odata-gateway.env`, chmod 600), мозг ходит через шлюз.

**Витрина SereneDB — read-only для бота:** отчёты идут под ролью `serene_ro` (GRANT SELECT), запись
БД физически запрещена (проверено: INSERT/UPDATE/DROP → permission denied). Сверх этого — валидатор
`serene_report`: LLM генерит только SELECT/WITH, любое изменяющее ключевое слово режется до выполнения.
Загрузчик витрины — отдельно, под rw.

🔴 **Анти-галлюцинации и анти-слив — КОДОМ (не промтом), на бот-слое:** `verify-плагин` OpenClaw
детерминированно (а) сверяет числа исходящего ответа с данными инструмента за этот ход — несверенное
режет/заменяет; (б) режет внутреннее (SQL, серверные пути, служебные маркеры) из текста И подписи
к фото. Промт (персона) держит только тон. Правило владельца: правила, которые должны ДЕРЖАТЬ, —
кодом. Детали — [`OPENCLAW_BOT.md`](OPENCLAW_BOT.md).

Сеть: OData не торчит в интернет — доступен только внутри доверенной сети LXC через роутер `.1`. Прод-усиление: HTTPS на IIS + firewall (только LXC) — см. §7.

Почему НЕ встроенный MCP-тулкит: он обслуживает HTTP через клиентский idle-обработчик 1С (~1 req/s, полностью встаёт при любом модальном окне сессии), и его UI-тумблеры не блокируют вызов инструментов. Детально из кода — [`TOOLKIT_TRANSPORT_ROOTCAUSE.md`](TOOLKIT_TRANSPORT_ROOTCAUSE.md). Артефакты MCP-пути (whitelist-прокси, форк) оставлены в репо как dev-опция.

## 5. Настройка под бизнес (разовая) vs автомат

Раскатка на новый бизнес = **разовая настройка**, дальше — само.

**Разовое (минуты):**
- Доступы + публикация OData на их 1С, `ai_reader`, шлюз, `.env` braine (их ключи/бот/KB-репо) — по [`RUNBOOK_DEPLOY.md`](RUNBOOK_DEPLOY.md).
- **Галочки «что тянуть»** (config-ui): открыл `http://<lxc>:6012` → отметил бизнес-разделы (продажи, склад, деньги, контрагенты…) → сохранил. Типовая 1С содержит сотни служебных классификаторов — их не отмечаем.
- (Опц.) golden-вопросы бизнеса для ночного регресса.

**Дальше автоматически:** ночной ETL по выбору → индексация → бот отвечает по свежим данным.

**Что универсально (код, копипаст на любой бизнес):** весь механизм — шлюз, ETL, config-ui, резолв ссылок guid→наименование, чистка полей. **Что конфиг (под бизнес):** список сущностей (галочки) + доступы/ключи в `.env`. Список — это конфигурация, а не хардкод в коде.

## 6. Zero-touch: что переживает ребут
- **Windows:** IIS (W3SVC) = Automatic; публикация/состав OData/пользователи персистентны.
- **LXC (system-сервисы `enabled`):** postgresql, open-webui, oikb, rerank-shim, api, kb-poll,
  1c-odata-gateway, 1c-config-ui, **serenedb**, **1c-mcp-braine**, **1c-mcp-reports** + таймеры
  (nightly-eval, 1c-etl, **1c-bot-monitor**). `tg-bridge` — **disabled** (Telegram у OpenClaw).
- **OpenClaw gateway** — systemd **user**-юнит юзера `undebot` с **linger=on** → стартует на буте
  без логина (проверено). Telegram — long polling, токен в `tokenFile`.
- Итог: перезагрузка любой машины → всё поднимается само; мониторинг алертит, если что-то не встало.

## 7. Тюнинг прода (не хвосты — работает; усилить на реальной ERP)
1. **Боевая ERP вместо стенда** — публикация OData на клиентской 1С:ERP, состав под их разделы, `ai_reader` по их правилам.
2. **Приватность LLM** — сейчас DeepSeek (ответы/NL→SQL) + Qwen `text-embedding-v4` (эмбеддинги, DashScope) облачные. Для реальных данных решить: компромисс (метаданные/агрегаты в облако) или self-host. У braine есть NER-анонимизация.
3. **Регистры** (остатки/обороты — «сколько на складе», «продажи за период») — в состав OData + `TOP_PREFIXES` ETL; в витрину SereneDB как факт-таблицы.
4. **Штатный инкремент-синк ETL→SereneDB** (сейчас POC-загрузчик разовый) + авто-пересборка `resolver_index`; инкремент документов по дате.
5. **HTTPS на IIS + firewall** (только LXC к OData); **ротация bot-токена** `@test1c_mcp_bot` перед реальным продом.
6. **Backup/restore SereneDB** (штатного нет — тест + вопрос фаундерам; витрина производная, пересобирается из 1С); бэкап pg OWUI; golden set под домен.
7. Этап 4 — миграция RAG braine на SereneDB (полнотекст+вектор в одном), вывод старого стека.
