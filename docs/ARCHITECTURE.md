# Архитектура «второго мозга» на данных 1С

Полное устройство системы: компоненты, потоки данных, безопасность, эксплуатация.
Как раскатать с нуля — [`RUNBOOK_DEPLOY.md`](RUNBOOK_DEPLOY.md).

Проверено end-to-end 2026-07-22..23 на стенде: Windows 11 + платформа 1С 8.3.27.1786 (x86) + Бухгалтерия 3.0 → OData/IIS → Ubuntu LXC (braine) → бот отвечает по данным 1С с цитатами.

---

## 1. Цель и инвариант

Дать AI-боту доступ к данным компании из 1С, чтобы он отвечал на вопросы бизнеса.

🔴 **Инвариант: 1С только читаем.** Ни один запрос не должен ничего изменить в 1С. Гарантия — слоями (§4), не на честном слове.

Замысел («второй мозг»): данные 1С регулярно выгружаются во внешнюю базу знаний и индексируются; бот отвечает по индексу с цитатами. 1С при этом не под нагрузкой «на каждый вопрос» и не под риском записи.

---

## 2. Топология (машины и сеть)

| Узел | Адрес | Роль |
|---|---|---|
| Windows-стенд | `10.8.0.58` (SSH), `192.168.122.141` (IIS) | 1С + IIS (OData). Только штатное ПО 1С, ничего кастомного. |
| Роутер сети LXC | `192.168.56.1` | Проброс порта → Windows-IIS:80 (соединение LXC↔1С по IP). |
| Ubuntu LXC | `192.168.56.42` | Весь наш код: шлюз, ETL, config-ui, braine. |

Сеть: LXC и Windows в разных подсетях; связь — через проброс порта на роутере `.1` (`192.168.56.1:<порт> → Windows-IIS:80`). Соединение по IP (без SSH-туннеля — встроенный сервер тулкита на туннеле зависал, а OData/IIS по IP работает штатно).

## 3. Компоненты и потоки данных

### Windows (штатная 1С, ничего кастомного)
- **1С:база** — файловая (стенд) или клиент-серверная (прод).
- **IIS** (служба Windows) публикует базу; включён штатный **OData** (`/1c/odata/standard.odata/`). Состав OData (какие объекты отдавать) задаётся один раз при публикации перебором всех справочников/документов.
- **Пользователь `ai_reader`** — роль «Только просмотр» (read-only). OData-запросы идут под ним (Basic auth).

### Ubuntu LXC (наш контроль)
| Сервис | Порт | Роль |
|---|---|---|
| `1c-odata-gateway` | 6011 | Read-only шлюз: пропускает только GET к OData; POST/PATCH/PUT/DELETE → 405. Хранит креды 1С (мозг их не знает). |
| `1c-config-ui` | 6012 | Веб-страница «галочки — что тянуть»: список сущностей 1С с числом записей, человек отмечает бизнес-разделы → `/etc/1c-etl-selected.txt`. |
| `1c-etl.timer/.service` | — | Ночью (03:00) выгружает выбранные сущности через шлюз → md-таблицы → пуш в KB-репо (GitLab). |
| `open-webui` | 3000 | Индекс/retrieval (эмбеддинги DashScope, pgvector). |
| `oikb` | 8081 | Синк KB-репо (GitLab) → Open WebUI. |
| `kb-poll` | — | Каждые 30с: новый коммит в KB-репо → триггер синка oikb. |
| `rerank-shim` | 8082 | Реранкер qwen3-rerank (DashScope) для retrieval. |
| `tg-bridge` | — | Telegram-бот: retrieval → гейты точности → DeepSeek → ответ с цитатами. |
| `api` | 8090 | HTTP API `POST /ask` (тот же пайплайн). |
| `postgresql` | 5432 | pgvector (вектора OWUI) + служебные схемы braine. |

### Холодный контур (наполнение знаний, ночью)
```
1С OData ──GET── шлюз :6011 ── ETL: выгрузка выбранных сущностей
   → md-таблицы (резолв ссылок guid→наименование, чистка полей)
   → git push в KB-репо (GitLab)
   → kb-poll видит коммит → oikb синкает → OpenWebUI индексирует (эмбеддинги) → pgvector
```

### Горячий контур (ответ на вопрос)
```
вопрос (Telegram / API) → tg-bridge/api → retrieval из индекса (гибрид BM25+вектор, реранк)
   → гейты точности (каждая цифра/дата сверяется с фрагментами, обязательные цитаты)
   → DeepSeek формулирует ответ из проверенного контекста → ответ с цитатами [n]
```
Движок горячего контура — **braine** (клонируемый RAG-шаблон): точность даёт код (гейты), а не промт; если данных нет — честное «нет данных». Разбор стека — [`UBUNTU_SETUP.md`](UBUNTU_SETUP.md).

## 4. Безопасность (read-only)

**Два независимых слоя** (для OData):
1. **Пользователь `ai_reader`** — прав записи нет: любая попытка записи через OData → отказ (проверено: чтение 200, запись 500).
2. **OData-шлюз** — deny-by-default по методу: только GET проходит; POST/PATCH/PUT/DELETE → 405, до 1С не доходят (проверено с LXC).

Плюс: **язык OData на чтение** (GET) не меняет данные по построению; состав OData ограничен опубликованными объектами; креды 1С — только в шлюзе (`/etc/1c-odata-gateway.env`, chmod 600), мозг ходит через шлюз.

Сеть: OData не торчит в интернет — доступен только внутри доверенной сети LXC через роутер `.1`. Прод-усиление: HTTPS на IIS + firewall (только LXC) — см. §7.

Почему НЕ встроенный MCP-тулкит: он обслуживает HTTP через клиентский idle-обработчик 1С (~1 req/s, полностью встаёт при любом модальном окне сессии), и его UI-тумблеры не блокируют вызов инструментов. Детально из кода — [`TOOLKIT_TRANSPORT_ROOTCAUSE.md`](TOOLKIT_TRANSPORT_ROOTCAUSE.md). Артефакты MCP-пути (whitelist-прокси, форк) оставлены в репо как dev-опция.

## 5. Настройка под бизнес (разовая) vs автомат

Раскатка на новый бизнес = **разовая настройка**, дальше — само.

**Разовое (минуты):**
- Доступы + публикация OData на их 1С, `ai_reader`, шлюз, `.env` braine (их ключи/бот/KB-репо) — по [`RUNBOOK_DEPLOY.md`](RUNBOOK_DEPLOY.md).
- **Галочки «что тянуть»** (config-ui): открыл `http://<lxc>:6012` → отметил бизнес-разделы (продажи, склад, деньги, контрагенты…) → сохранил. Типовая 1С содержит сотни служебных классификаторов — их не отмечаем.
- (Опц.) golden-вопросы бизнеса для ночного регресса.

**Дальше автоматически:** ночной ETL по выбору → индексация → бот отвечает по свежим данным.

**Что универсально (код, копипаст на любой бизнес):** весь механизм — шлюз, ETL, config-ui, резолв ссылок guid→наименование, чистка полей. **Что конфиг (под бизнес):** список сущностей (галочки) + доступы/ключи в `.env`. Список — это конфигурация, а не хардкод в коде.

## 6. Zero-touch: что переживает ребут
- **Windows:** IIS (W3SVC) = Automatic; публикация/состав OData/пользователи персистентны.
- **LXC:** все сервисы `enabled` (postgresql, open-webui, oikb, rerank-shim, tg-bridge, api, kb-poll, 1c-odata-gateway, 1c-config-ui) + таймеры (nightly-eval, 1c-etl).
- Итог: перезагрузка любой машины → всё поднимается само, без ручных действий.

## 7. Тюнинг прода (не хвосты — работает; усилить на реальной ERP)
1. **Боевая ERP вместо стенда** — публикация OData на клиентской 1С:ERP, состав под их разделы, `ai_reader` по их правилам.
2. **Приватность LLM** — сейчас DeepSeek (ответы) + DashScope (эмбеддинги) облачные. Для реальных данных решить: компромисс (метаданные/агрегаты в облако) или self-host LLM. У braine есть NER-анонимизация.
3. **Регистры** (остатки/обороты — «сколько на складе», «продажи за период») — добавить в состав OData + `TOP_PREFIXES` в ETL.
4. **Инкремент документов по дате** (сейчас полная перевыгрузка; идемпотентно).
5. **HTTPS на IIS + firewall** (только LXC к OData); ротация токенов.
6. **Golden set под домен клиента**; мониторинг/алерты; бэкап pg OWUI по расписанию.
